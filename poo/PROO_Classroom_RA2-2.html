Sección 2 · Patrones de diseño

¿Qué es un patrón de diseño?

Un patrón de diseño es una solución probada y reutilizable a un problema que aparece una y otra vez al programar. No es código que copias literal, sino una "receta" o plantilla que adaptas a tu caso.

Una analogía del mundo real

Cuando un arquitecto diseña una casa, no inventa desde cero cómo hacer una escalera: usa patrones conocidos (escalera recta, en L, de caracol) según el espacio. En programación es igual: ante un problema común (¿cómo garantizar que solo exista un objeto?, ¿cómo crear objetos sin saber su tipo exacto?), aplicas un patrón ya probado por miles de programadores.

Los patrones que verás en este RA

El programa PROO-20 se enfoca en patrones útiles para la autenticación en aplicaciones (como el login de un cajero):

Patrón¿Qué problema resuelve?
SingletonGarantizar que exista UNA sola instancia de una clase (ej. un único gestor de sesión).
Factory MethodCrear objetos sin especificar su clase exacta (ej. crear el tipo de cuenta correcto según un parámetro).
Abstract FactoryCrear familias de objetos relacionados (ej. todo un conjunto de componentes de una interfaz).
PrototypeCrear objetos nuevos copiando ("clonando") uno existente.
¿Por qué son importantes para la seguridad?

En un sistema de autenticación, los patrones ayudan a que el control de acceso sea consistente y centralizado. Por ejemplo, el Singleton evita que existan varios gestores de sesión descoordinados (un hueco de seguridad), y la Factory asegura que cada usuario reciba exactamente el tipo de cuenta y permisos que le corresponden.

Sección 3 · Patrones de diseño

Patrón Singleton

El Singleton garantiza que una clase tenga una sola instancia en todo el programa, y ofrece un punto de acceso global a ella. Perfecto para cosas que deben ser únicas: el gestor de sesión del cajero, la conexión a la base de datos, la configuración del sistema.

El problema que resuelve

Imagina que en el cajero hubiera varios gestores de sesión a la vez: uno diría que el usuario está autenticado, otro que no. Un caos de seguridad. El Singleton asegura que solo exista uno, y que todos consulten ese mismo.

Cómo se construye

Tres ingredientes: (1) constructor privado (nadie puede crear objetos desde afuera), (2) un atributo estático que guarda la única instancia, y (3) un método público getInstancia() que la devuelve (creándola la primera vez).

#include <iostream>
using namespace std;

class GestorSesion {
  private:
    static GestorSesion* instancia;   // la única instancia
    GestorSesion() {                  // constructor PRIVADO
        cout << "Gestor de sesión creado" << endl;
    }
  public:
    static GestorSesion* getInstancia() {
        if (instancia == nullptr) {   // si no existe, la crea
            instancia = new GestorSesion();
        }
        return instancia;             // siempre devuelve la misma
    }
    void autenticar(string usuario) {
        cout << "Autenticando a " << usuario << endl;
    }
};

// Inicializar el puntero estático (obligatorio, fuera de la clase)
GestorSesion* GestorSesion::instancia = nullptr;

int main() {
    // No se puede: GestorSesion g;  ❌ (constructor privado)
    GestorSesion* g1 = GestorSesion::getInstancia();  // crea
    GestorSesion* g2 = GestorSesion::getInstancia();  // reusa la misma
    g1->autenticar("Ana");

    // g1 y g2 son EL MISMO objeto:
    cout << (g1 == g2 ? "Misma instancia" : "Distintas") << endl;
    return 0;
}
Ingrediente¿Para qué?
Constructor privateImpide que se creen objetos con GestorSesion g; desde afuera.
static ... instanciaGuarda la única instancia, compartida por todo el programa.
getInstancia()Punto de acceso: crea la instancia la primera vez, la reutiliza después.
La clave del Singleton

La salida mostrará "Gestor de sesión creado" una sola vez, aunque llamemos getInstancia() dos veces. Y g1 == g2 es verdadero: son el mismo objeto. Eso es exactamente lo que el patrón garantiza.

Sección 4 · Patrones de diseño

Patrón Factory Method

El Factory Method (Método de Fábrica) crea objetos sin que tengas que escribir directamente cuál clase. Le pides a una "fábrica" el objeto que necesitas pasándole un parámetro, y ella decide qué crear. Ideal para crear distintos tipos de cuenta.

El problema que resuelve

Sin el patrón, tu código estaría lleno de if: "si es ahorro, crea CuentaAhorro; si es crédito, crea CuentaCredito..." repetido en muchos lugares. La fábrica centraliza esa decisión en un solo lugar. Si mañana agregas un tipo nuevo, solo cambias la fábrica.

Cómo se construye

#include <iostream>
using namespace std;

// Clase base (producto)
class Cuenta {
  public:
    virtual void tipo() { cout << "Cuenta genérica" << endl; }
};

class CuentaAhorro : public Cuenta {
  public:
    void tipo() override { cout << "Cuenta de Ahorro" << endl; }
};

class CuentaCredito : public Cuenta {
  public:
    void tipo() override { cout << "Cuenta de Crédito" << endl; }
};

// LA FÁBRICA: decide qué objeto crear según el parámetro
class FabricaCuentas {
  public:
    static Cuenta* crear(string tipo) {
        if (tipo == "ahorro")  return new CuentaAhorro();
        if (tipo == "credito") return new CuentaCredito();
        return new Cuenta();
    }
};

int main() {
    // No decimos "new CuentaAhorro": la fábrica decide
    Cuenta* c1 = FabricaCuentas::crear("ahorro");
    Cuenta* c2 = FabricaCuentas::crear("credito");
    c1->tipo();   // Cuenta de Ahorro
    c2->tipo();   // Cuenta de Crédito
    return 0;
}
La idea central

El main no sabe (ni le importa) qué clase exacta se crea — solo pide "ahorro" o "credito" y la fábrica entrega el objeto correcto. Esto se combina perfecto con el polimorfismo del RA 2.1: la fábrica devuelve un Cuenta* y cada objeto responde con su propio tipo().

Sección 5 · Patrones de diseño

Abstract Factory y Prototype

Dos patrones más que pide el programa. Los vemos a nivel conceptual con ejemplos breves: Abstract Factory (fábrica de familias de objetos) y Prototype (crear copiando).

Abstract Factory (Fábrica Abstracta)

Si el Factory Method crea un producto, la Abstract Factory crea familias completas de productos relacionados. Imagina que el cajero tiene dos "estilos": clientes normales y clientes premium. Cada estilo necesita su propia tarjeta Y su propio comprobante, que deben combinar entre sí.

// Una fábrica abstracta produce una familia coherente:
class FabricaCliente {
  public:
    virtual string crearTarjeta() = 0;       // método virtual puro
    virtual string crearComprobante() = 0;
};

class FabricaPremium : public FabricaCliente {
  public:
    string crearTarjeta() override { return "Tarjeta Oro"; }
    string crearComprobante() override { return "Comprobante Premium"; }
};

class FabricaNormal : public FabricaCliente {
  public:
    string crearTarjeta() override { return "Tarjeta Clásica"; }
    string crearComprobante() override { return "Comprobante Normal"; }
};
La diferencia clave

Factory Method: crea UN objeto (una cuenta). Abstract Factory: crea una FAMILIA de objetos que van juntos (tarjeta + comprobante del mismo estilo). Garantiza que no mezcles una "Tarjeta Oro" con un "Comprobante Normal".

Prototype (Prototipo)

El patrón Prototype crea objetos nuevos clonando uno existente, en lugar de construirlos desde cero. Útil cuando crear un objeto es costoso o cuando quieres copiar una configuración ya lista.

class Cuenta {
  public:
    string titular;
    double saldo;

    // Método clonar: devuelve una copia de sí mismo
    Cuenta* clonar() {
        Cuenta* copia = new Cuenta();
        copia->titular = this->titular;
        copia->saldo = this->saldo;
        return copia;
    }
};

int main() {
    Cuenta original;
    original.titular = "Plantilla";
    original.saldo = 0;

    Cuenta* nueva = original.clonar();  // copia basada en el prototipo
    nueva->titular = "Ana";
    return 0;
}
¿Cuándo usar Prototype?

Cuando tienes un objeto "plantilla" ya configurado y quieres crear muchos similares cambiando solo unos detalles. En vez de configurar cada uno desde cero, clonas el prototipo y ajustas lo necesario. this se refiere al objeto actual.

Sección 6 · Operación continua

Cifrado y ciclo de vida de los objetos

Para que el sistema opere de forma continua y segura, hay que proteger los datos sensibles (cifrado) y manejar bien el nacimiento y muerte de los objetos (inicialización y destrucción).

Cifrado de datos

Cifrar es transformar un dato legible en uno ilegible para quien no tenga la clave. En el cajero, el NIP nunca debe guardarse "en claro". Un cifrado sencillo (didáctico) es el César: desplazar cada carácter un número fijo de posiciones.

#include <iostream>
using namespace std;

class Seguridad {
  public:
    // Cifrado César simple: desplaza cada carácter +3
    string cifrar(string texto) {
        string resultado = texto;
        for (int i = 0; i < texto.length(); i++) {
            resultado[i] = texto[i] + 3;
        }
        return resultado;
    }
    string descifrar(string texto) {
        string resultado = texto;
        for (int i = 0; i < texto.length(); i++) {
            resultado[i] = texto[i] - 3;
        }
        return resultado;
    }
};

int main() {
    Seguridad s;
    string nip = "1234";
    string cifrado = s.cifrar(nip);
    cout << "Cifrado: " << cifrado << endl;
    cout << "Descifrado: " << s.descifrar(cifrado) << endl;
    return 0;
}
⚠ Nota importante

El cifrado César es solo didáctico para entender el concepto — en sistemas reales se usan algoritmos robustos (AES, bcrypt). Pero la idea es la misma: el dato sensible nunca se guarda ni viaja en texto legible.

Inicialización y destrucción de objetos

Recuerda del RA 1.2: el constructor inicializa el objeto (reserva recursos, abre conexiones) y el destructor lo limpia al morir (libera memoria, cierra conexiones). Para la operación continua, esto es vital: un objeto que no libera sus recursos provoca "fugas" que con el tiempo tumban el sistema.

class Conexion {
  public:
    Conexion() {                          // INICIALIZACIÓN
        cout << "Conexión abierta" << endl;
    }
    ~Conexion() {                         // DESTRUCCIÓN
        cout << "Conexión cerrada (recursos liberados)" << endl;
    }
};

int main() {
    Conexion c;     // se abre aquí
    cout << "Usando la conexión..." << endl;
    return 0;       // el destructor se ejecuta solo al terminar
}
// Salida:
//   Conexión abierta
//   Usando la conexión...
//   Conexión cerrada (recursos liberados)
¿Por qué importa para la operación continua?

Si cada objeto libera correctamente sus recursos al destruirse (memoria, archivos, conexiones), el sistema puede funcionar por días sin degradarse. Objetos mal destruidos = fugas de memoria = el cajero se vuelve lento y eventualmente falla. El destructor bien hecho es seguro de operación.

Sección 7 · Manejo de excepciones

Manejo de excepciones

Una excepción es un error que ocurre durante la ejecución (retirar más de lo que hay, dividir entre cero, datos inválidos). El manejo de excepciones permite atrapar esos errores y responder con elegancia, en lugar de que el programa se caiga.

try, throw, catch

Tres palabras clave: try (intenta este código), throw (lanza un error si algo va mal) y catch (atrapa el error y lo maneja).

#include <iostream>
using namespace std;

class Cuenta {
  private:
    double saldo = 1000;
  public:
    void retirar(double monto) {
        if (monto > saldo) {
            throw "Fondos insuficientes";   // LANZA el error
        }
        saldo -= monto;
        cout << "Retiro exitoso. Saldo: " << saldo << endl;
    }
};

int main() {
    Cuenta c;
    try {
        c.retirar(500);    // funciona
        c.retirar(2000);   // lanza excepción
        c.retirar(100);    // ya no se ejecuta
    }
    catch (const char* error) {            // ATRAPA el error
        cout << "Error capturado: " << error << endl;
    }
    cout << "El programa continúa sin caerse." << endl;
    return 0;
}
Palabra clave¿Qué hace?
try { }Envuelve el código que podría fallar.
throwLanza el error cuando se detecta una condición inválida.
catch { }Atrapa el error lanzado y ejecuta el plan B (mensaje, registro, recuperación).
¿Por qué es clave para la seguridad y la operación continua?

Sin manejo de excepciones, un retiro inválido o un dato corrupto tumbaría todo el cajero. Con try/catch, el error se atrapa, se registra, se le avisa al usuario, y el sistema sigue funcionando para el siguiente cliente. Eso es robustez: fallar de forma controlada, nunca catastrófica.

Sección 8 · Ejemplo guiado

Cajero seguro: patrones + excepciones

Integramos todo: un Singleton para el gestor de sesión, una Factory para crear cuentas, cifrado del NIP y excepciones para los errores. Léelo con calma: cada pieza ya la viste.

El programa completo

#include <iostream>
using namespace std;

// ---- SINGLETON: un único gestor de sesión ----
class GestorSesion {
  private:
    static GestorSesion* instancia;
    GestorSesion() {}
  public:
    static GestorSesion* getInstancia() {
        if (instancia == nullptr) instancia = new GestorSesion();
        return instancia;
    }
    string cifrarNIP(string nip) {          // cifrado César
        for (int i = 0; i < nip.length(); i++) nip[i] = nip[i] + 3;
        return nip;
    }
};
GestorSesion* GestorSesion::instancia = nullptr;

// ---- FACTORY: crea el tipo de cuenta correcto ----
class Cuenta {
  public:
    virtual void tipo() { cout << "Genérica" << endl; }
};
class CuentaAhorro : public Cuenta {
  public:
    void tipo() override { cout << "Ahorro" << endl; }
};
class FabricaCuentas {
  public:
    static Cuenta* crear(string t) {
        if (t == "ahorro") return new CuentaAhorro();
        return new Cuenta();
    }
};

int main() {
    // Singleton: una sola sesión
    GestorSesion* sesion = GestorSesion::getInstancia();
    cout << "NIP cifrado: " << sesion->cifrarNIP("1234") << endl;

    // Factory: crea la cuenta
    Cuenta* c = FabricaCuentas::crear("ahorro");
    c->tipo();

    // Excepciones: maneja un error sin caerse
    try {
        double monto = 5000, saldo = 1000;
        if (monto > saldo) throw "Fondos insuficientes";
    }
    catch (const char* e) {
        cout << "Error manejado: " << e << endl;
    }
    cout << "Sistema operando con normalidad." << endl;
    return 0;
}

¿Ves cómo todo encaja?

Concepto del RA 2.2Dónde aparece
SingletonGestorSesion con constructor privado y getInstancia()
Factory MethodFabricaCuentas::crear() decide qué cuenta construir
Cifrado de datoscifrarNIP() protege el NIP con desplazamiento César
Manejo de excepcionestry/throw/catch atrapa el error sin tumbar el sistema

Entrega tu reporte

Cuando estés conforme con tu resultado, envíalo a tu docente con el botón azul. El envío es único y definitivo: tu calificación llega directamente al correo del docente con fecha, hora y folio, y el cuaderno queda sellado — las actividades y la bitácora pasan a solo lectura. Los apuntes, las instrucciones de cada práctica y el editor de C++ seguirán disponibles para consulta.

Antes de enviar, verifica

Que tus 4 actividades estén entregadas y tu bitácora guardada. Después del envío ya no podrás modificar nada que genere calificación. Tu docente recibe cada envío con folio y hora: los envíos duplicados desde otro equipo se detectan y solo cuenta el primero.
