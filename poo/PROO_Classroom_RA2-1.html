Sección 1 · Introducción

Herencia, sobrecarga y polimorfismo

En el R.A. 1.2 aprendiste a crear clases y objetos. Ahora subimos de nivel: harás que unas clases hereden de otras (reutilizando código), que tus objetos respondan diferente al mismo mensaje (polimorfismo), y que los operadores como + funcionen con tus propios objetos (sobrecarga).

Los tres grandes temas de este RA

  • Herencia: una clase "hija" reutiliza y extiende lo de una clase "padre". Una CuentaAhorro es una Cuenta que además gana intereses.
  • Sobrecarga de operadores: darle significado a +, ==, etc. para tus objetos. Sumar dos cuentas con c1 + c2.
  • Polimorfismo: el mismo método (mostrarTipo()) se comporta distinto según el tipo real del objeto.

Seguimos con el Cajero Automático 🏧

Nuestro cajero crece: ahora maneja distintos tipos de cuenta (ahorro, crédito, nómina). Aquí es donde la herencia y el polimorfismo brillan — en lugar de copiar y pegar código para cada tipo, defines una vez lo común y cada tipo agrega solo lo suyo.

Contexto del Resultado de Aprendizaje

R.A. 2.1 (programa PROO-20): "Elabora aplicaciones mediante la interacción de los objetos y actores del sistema" (60 horas). Tu evidencia (Actividad de evaluación 2.1.1, 35% del módulo) es la codificación de aplicaciones usando herencia y polimorfismo.

🏧 Práctica con código real

Igual que en el repaso, vas a escribir y ejecutar C++ de verdad en el navegador. La herencia y el polimorfismo se entienden mucho mejor cuando ves la salida del programa con tus propios ojos.

Sección 2 · Repaso

Clases: lo que ya dominas

La herencia se construye sobre las clases. Repaso exprés de lo esencial del R.A. 1.2 antes de subir de nivel.

Una clase con atributos privados, constructor y métodos

#include <iostream>
using namespace std;

class Cuenta {
  protected:                 // ¡ojo! protected en vez de private
    string titular;
    double saldo;
  public:
    Cuenta(string n, double s) {
        titular = n;
        saldo = s;
    }
    void depositar(double m) { saldo += m; }
    void mostrar() {
        cout << titular << ": $" << saldo << endl;
    }
};
🔑 La pieza clave para la herencia: protected

¿Recuerdas los tres niveles de acceso? private (solo la clase), public (todos) y protected. Este último era "un misterio" en el R.A. 1.2 — ahora cobra sentido: protected permite que las clases hijas accedan al atributo, pero nadie de afuera. Es el nivel perfecto para herencia: el saldo está protegido del exterior, pero CuentaAhorro (la hija) sí puede usarlo.

Crear y usar objetos

int main() {
    Cuenta c("Ana", 1000.0);
    c.depositar(500);
    c.mostrar();        // Ana: $1500
    return 0;
}
Sección 3 · Herencia

Herencia y reutilización de código

La herencia permite crear una clase nueva (hija) a partir de una existente (padre), heredando sus atributos y métodos. La hija reutiliza lo que ya existe y agrega o cambia lo que necesita.

La idea: "es un/una"

Una CuentaAhorro es una Cuenta (con algo extra: intereses). Una CuentaCredito es una Cuenta (con límite de crédito). En lugar de repetir el código de titular, saldo, depositar... lo heredan de Cuenta.

Sintaxis de la herencia

class Cuenta {            // CLASE PADRE (base)
  protected:
    string titular;
    double saldo;
  public:
    Cuenta(string n, double s) { titular = n; saldo = s; }
    void mostrar() { cout << titular << ": $" << saldo << endl; }
};

class CuentaAhorro : public Cuenta {   // HIJA hereda de Cuenta
  private:
    double tasaInteres;
  public:
    // El constructor de la hija llama al del padre con ": Cuenta(...)"
    CuentaAhorro(string n, double s, double tasa) : Cuenta(n, s) {
        tasaInteres = tasa;
    }
    // MÉTODO NUEVO, propio de la hija:
    void aplicarInteres() {
        saldo += saldo * tasaInteres;   // usa 'saldo' heredado (protected)
    }
};
Parte¿Qué significa?
class CuentaAhorro : public Cuenta"CuentaAhorro hereda públicamente de Cuenta". Recibe todos sus miembros.
: Cuenta(n, s)El constructor de la hija llama al constructor del padre para inicializar lo heredado.
saldo += ...La hija usa saldo, que heredó del padre (por eso era protected).

Usar la clase hija

int main() {
    CuentaAhorro ca("Ana", 1000.0, 0.10);  // 10% de interés
    ca.mostrar();         // Ana: $1000   (método HEREDADO de Cuenta)
    ca.aplicarInteres();  // método PROPIO de CuentaAhorro
    ca.mostrar();         // Ana: $1100
    return 0;
}

Fíjate: ca puede usar mostrar() (heredado) y aplicarInteres() (propio). Eso es reutilización: no reescribimos mostrar().

Dos beneficios clave

  • Reutilización de código: el código común (titular, saldo, depositar, mostrar) se escribe UNA vez en el padre y todas las hijas lo aprovechan.
  • Extensibilidad: si mañana necesitas una CuentaNomina, solo creas la hija con lo nuevo — sin tocar lo que ya funciona. El sistema crece sin romperse.
Sección 4 · Herencia

Tipos de herencia y ambigüedad

La herencia puede ser simple (una hija, un padre) o múltiple (una hija, varios padres). La múltiple es poderosa pero trae un riesgo: la ambigüedad.

Herencia simple

Es la que ya viste: una clase hereda de un solo padre. CuentaAhorro : public Cuenta. Es la más común y la más clara.

class Cuenta { /* ... */ };
class CuentaAhorro : public Cuenta { /* ... */ };   // 1 padre

Herencia múltiple

Una clase hereda de dos o más padres a la vez, combinando sus capacidades. Por ejemplo, una CuentaInversion que es a la vez Cuenta (maneja saldo) y Reportable (sabe generar reportes):

class Cuenta {
  protected:
    double saldo;
  public:
    Cuenta(double s) { saldo = s; }
    void mostrarSaldo() { cout << "Saldo: " << saldo << endl; }
};

class Reportable {
  public:
    void generarReporte() { cout << "Generando reporte..." << endl; }
};

// Hereda de DOS padres, separados por coma:
class CuentaInversion : public Cuenta, public Reportable {
  public:
    CuentaInversion(double s) : Cuenta(s) {}
};

int main() {
    CuentaInversion ci(5000);
    ci.mostrarSaldo();      // viene de Cuenta
    ci.generarReporte();    // viene de Reportable
    return 0;
}

El problema de la ambigüedad

¿Qué pasa si los dos padres tienen un método con el mismo nombre? El compilador no sabe cuál usar — eso es la ambigüedad.

class Cuenta {
  public:
    void info() { cout << "Info de cuenta" << endl; }
};
class Reportable {
  public:
    void info() { cout << "Info de reporte" << endl; }  // ¡mismo nombre!
};
class CuentaInversion : public Cuenta, public Reportable { };

int main() {
    CuentaInversion ci;
    // ci.info();          // ❌ ERROR: ambiguo, ¿cuál info()?
    ci.Cuenta::info();      // ✓ se resuelve indicando la clase
    ci.Reportable::info();  // ✓ con el operador de ámbito ::
    return 0;
}
⚠ Cómo resolver la ambigüedad

Se usa el operador de resolución de ámbito :: para decirle al compilador de qué clase quieres el método: ci.Cuenta::info(). Por eso muchos lenguajes evitan la herencia múltiple — C++ la permite, pero hay que usarla con cuidado.

Sección 5 · Polimorfismo

Sobrecarga de operadores

Normalmente + suma números. Pero, ¿y si quieres sumar dos cuentas, o comparar dos cuentas con ==? La sobrecarga de operadores te deja darle un nuevo significado a los operadores para que funcionen con tus propios objetos.

La idea

Sobrecargar un operador es escribir un método especial llamado operator seguido del símbolo. Cuando C++ ve c1 + c2 entre objetos, llama a tu método operator+.

Operador binario: + (dos operandos)

Un operador binario trabaja con dos valores (izquierda y derecha). Vamos a sumar los saldos de dos cuentas:

class Cuenta {
  public:
    double saldo;
    Cuenta(double s) { saldo = s; }

    // Sobrecarga del operador +
    Cuenta operator+(Cuenta otra) {
        return Cuenta(saldo + otra.saldo);
    }
};

int main() {
    Cuenta c1(1000), c2(500);
    Cuenta total = c1 + c2;     // llama a operator+
    cout << total.saldo;        // 1500
    return 0;
}
Parte¿Qué significa?
operator+El nombre especial del método que sobrecarga el símbolo +.
Cuenta otraEl operando de la derecha (c2). El de la izquierda (c1) es el objeto actual.
return Cuenta(...)Devuelve una nueva cuenta con la suma de los saldos.

Operador unario: ++ (un operando)

Un operador unario trabaja con un solo valor. Por ejemplo, ++ para "premiar" una cuenta sumándole un bono fijo de $100:

class Cuenta {
  public:
    double saldo;
    Cuenta(double s) { saldo = s; }

    // Sobrecarga del operador ++ (unario)
    void operator++() {
        saldo += 100;     // bono de $100
    }
};

int main() {
    Cuenta c(1000);
    ++c;                  // llama a operator++
    cout << c.saldo;      // 1100
    return 0;
}
Binario vs unario

Binario: necesita dos operandos (c1 + c2, c1 == c2). Su método recibe el otro objeto como parámetro. Unario: opera sobre un solo objeto (++c, -c). Su método no recibe parámetros (trabaja sobre sí mismo).

Sección 6 · Polimorfismo

Polimorfismo y funciones virtuales

Polimorfismo significa "muchas formas": el mismo método se comporta diferente según el tipo real del objeto. Un mensaje (mostrarTipo()) produce respuestas distintas en una cuenta de ahorro, una de crédito o una de nómina.

Polimorfismo en funciones: sobrescribir métodos

Cuando una clase hija define un método con el mismo nombre que el padre, lo sobrescribe (override). El objeto hijo usa su propia versión:

class Cuenta {
  public:
    void mostrarTipo() {
        cout << "Cuenta genérica" << endl;
    }
};

class CuentaAhorro : public Cuenta {
  public:
    void mostrarTipo() {        // sobrescribe el del padre
        cout << "Cuenta de Ahorro" << endl;
    }
};

int main() {
    CuentaAhorro ca;
    ca.mostrarTipo();           // Cuenta de Ahorro (su versión)
    return 0;
}

El verdadero poder: funciones virtuales

El polimorfismo brilla cuando manejas objetos de distintos tipos a través de un puntero al padre. La palabra clave virtual le dice a C++: "decide en tiempo de ejecución cuál versión usar, según el tipo REAL del objeto".

class Cuenta {
  public:
    virtual void mostrarTipo() {     // ¡virtual!
        cout << "Cuenta genérica" << endl;
    }
};

class CuentaAhorro : public Cuenta {
  public:
    void mostrarTipo() override {
        cout << "Cuenta de Ahorro" << endl;
    }
};

class CuentaCredito : public Cuenta {
  public:
    void mostrarTipo() override {
        cout << "Cuenta de Crédito" << endl;
    }
};

int main() {
    Cuenta* cuentas[2];               // arreglo de punteros al PADRE
    cuentas[0] = new CuentaAhorro();
    cuentas[1] = new CuentaCredito();

    for (int i = 0; i < 2; i++) {
        cuentas[i]->mostrarTipo();    // cada uno usa SU versión
    }
    // Salida:
    //   Cuenta de Ahorro
    //   Cuenta de Crédito
    return 0;
}
⭐ ¿Por qué virtual es tan importante?

Sin virtual, al usar un puntero Cuenta* el programa siempre llamaría la versión del padre (decide por el tipo del puntero). Con virtual, C++ mira el tipo real del objeto en tiempo de ejecución y llama la versión correcta. Eso permite tratar muchos tipos distintos de forma uniforme — el corazón del polimorfismo.

El flujo completo de la POO

Herencia (CuentaAhorro hereda de Cuenta) + sobrescritura (cada hija define su mostrarTipo) + virtual (decisión en tiempo de ejecución) = un cajero que maneja cualquier tipo de cuenta con el mismo código. Agregar un tipo nuevo no rompe nada: pura extensibilidad.

Sección 7 · Ejemplo guiado

El cajero con varios tipos de cuenta

Juntamos todo: una clase base Cuenta, dos hijas que heredan y sobrescriben, y un main que las maneja polimórficamente. Léelo con calma: cada pieza ya la viste.

El programa completo

#include <iostream>
using namespace std;

// CLASE BASE
class Cuenta {
  protected:
    string titular;
    double saldo;
  public:
    Cuenta(string n, double s) {
        titular = n;
        saldo = s;
    }
    virtual void aplicarBeneficio() {     // virtual: las hijas lo cambian
        cout << "Cuenta sin beneficios" << endl;
    }
    void mostrar() {
        cout << titular << ": $" << saldo << endl;
    }
};

// HIJA 1: gana intereses
class CuentaAhorro : public Cuenta {
  public:
    CuentaAhorro(string n, double s) : Cuenta(n, s) {}
    void aplicarBeneficio() override {
        saldo += saldo * 0.10;            // 10% de interés
        cout << "Interés aplicado a " << titular << endl;
    }
};

// HIJA 2: recibe un bono fijo
class CuentaNomina : public Cuenta {
  public:
    CuentaNomina(string n, double s) : Cuenta(n, s) {}
    void aplicarBeneficio() override {
        saldo += 200;                     // bono de nómina
        cout << "Bono aplicado a " << titular << endl;
    }
};

int main() {
    // Un arreglo de punteros al PADRE, con objetos de distintas hijas
    Cuenta* cuentas[2];
    cuentas[0] = new CuentaAhorro("Ana", 1000);
    cuentas[1] = new CuentaNomina("Luis", 800);

    // POLIMORFISMO: el mismo bucle, cada cuenta hace lo suyo
    for (int i = 0; i < 2; i++) {
        cuentas[i]->mostrar();
        cuentas[i]->aplicarBeneficio();   // virtual: versión correcta
        cuentas[i]->mostrar();
        cout << "---" << endl;
    }
    return 0;
}

¿Ves cómo todo encaja?

Concepto del RA 2.1Dónde aparece
HerenciaCuentaAhorro y CuentaNomina heredan de Cuenta (titular, saldo, mostrar)
Constructor de la hija que llama al padreCuentaAhorro(...) : Cuenta(n, s)
Sobrescritura de métodoscada hija redefine aplicarBeneficio()
