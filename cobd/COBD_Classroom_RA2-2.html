Sección 1 · Introducción

De los datos crudos a la información presentable

En el RA 2.1 aprendiste a manipular los datos. Pero a la dirección del plantel no le sirve una tabla cruda: necesita vistas reutilizables, formularios para capturar sin errores y reportes listos para tomar decisiones. Eso es exactamente el RA 2.2 — la cara presentable de tu base de datos.

Las tres herramientas de este RA

  • Vistas (VIEW): consultas guardadas con nombre. Escribes una vez la consulta compleja (con sus JOIN y filtros) y después cualquiera la usa como si fuera una tabla sencilla.
  • Formularios: la interfaz amigable de captura. El usuario llena campos y pica un botón; por detrás, el formulario ejecuta el INSERT o UPDATE que tú ya dominas.
  • Reportes: consultas con presentación: columnas renombradas (AS), datos ordenados (ORDER BY) y resúmenes calculados (COUNT, AVG, SUM) listos para entregarse.

¿Dónde lo ves en la vida real?

  • Cuando control escolar imprime tu boleta, está ejecutando un reporte sobre una vista con JOIN de alumnos, materias y calificaciones.
  • Cuando te inscribes en línea, llenas un formulario que ejecuta un INSERT validado.
  • Cuando la dirección revisa el promedio por grupo, lee un reporte con GROUP BY y AVG.

Sección 2 · Repaso · RA 2.1

SQL exprés: lo que ya dominas (y vas a necesitar)

Las vistas y los reportes se construyen encima de las consultas del RA 2.1. Repaso de 3 minutos antes de subir de nivel.

Tu caja de herramientas del RA 2.1

🗄 SistemaAdministracionEstudiantil › alumnos
-- Consultar y filtrar
SELECT nombre, promedio FROM alumnos WHERE grupo = '410' AND promedio >= 8.5;

-- Mantener los datos
INSERT INTO alumnos (matricula, nombre, apellido_paterno, grupo, turno, promedio)
VALUES ('24100501', 'Diana', 'Cervantes', '410', 'Matutino', 0.0);
UPDATE alumnos SET turno = 'Vespertino' WHERE matricula = '23100459';
DELETE FROM alumnos WHERE matricula = '23100378';
🗄 SistemaAdministracionEstudiantil › alumnos + materias + calificaciones
-- Combinar tablas (la base de las vistas que crearás)
SELECT a.nombre, m.nombre_materia, c.calificacion
FROM calificaciones c
INNER JOIN alumnos  a ON c.matricula  = a.matricula
INNER JOIN materias m ON c.id_materia = m.id_materia;
Novedad de sintaxis: alias de tabla

¿Notaste las letras c, a y m? Son alias de tabla: apodos cortos que le pones a cada tabla justo después de su nombre (FROM calificaciones c) para no escribir el nombre completo en cada columna. c.matricula y calificaciones.matricula significan exactamente lo mismo — el alias solo ahorra tecleo. En el RA 2.1 escribías los nombres completos; a partir de aquí usaremos alias, como se hace en los proyectos reales.

⚠ Sigue vigente la regla de oro

UPDATE y DELETE siempre con WHERE, filtrando por la llave (matricula). Los formularios profesionales lo hacen exactamente así por detrás.

Sección 3 · Conceptos · Contenidos A y B del RA 2.2

Vistas, formularios y reportes

Tres herramientas, un mismo objetivo: que la información de la base llegue a las personas correctas, en el formato correcto, sin que nadie tenga que escribir SQL desde cero cada vez.

A · Vistas (VIEW) — consultas guardadas con nombre

Una vista es una consulta SELECT que el gestor guarda con un nombre. A partir de ahí, la consultas como si fuera una tabla, pero sin duplicar datos: cada vez que la usas, el gestor ejecuta por dentro la consulta original sobre las tablas reales, así que siempre muestra información actualizada.

Crear una vista

🗄 SistemaAdministracionEstudiantil › vista (alumnos)
-- Sintaxis: CREATE VIEW nombre AS consulta
CREATE VIEW vista_matutino AS
SELECT nombre, apellido_paterno, grupo, promedio
FROM alumnos
WHERE turno = 'Matutino';

Usar una vista

-- Se consulta igual que una tabla (y acepta WHERE, ORDER BY...)
SELECT * FROM vista_matutino;
SELECT nombre, promedio FROM vista_matutino WHERE promedio >= 9;

Eliminar una vista

-- Borra la vista, NUNCA los datos: las tablas originales quedan intactas
DROP VIEW vista_matutino;
¿Para qué molestarse en crear vistas?

1) Simplicidad: la consulta compleja (con 2 JOIN y filtros) se escribe UNA vez; después todos usan SELECT * FROM vista_boletas. 2) Seguridad: puedes dar acceso a la vista sin exponer columnas delicadas de la tabla. 3) Consistencia: todos los reportes parten de la misma definición.

B · Formularios — la capa amable de captura

Un formulario es la interfaz con campos y botones que ve el usuario. El secretario de control escolar no escribe SQL: llena cajas de texto y pica "Guardar". Pero por detrás, ese botón ejecuta exactamente las instrucciones que tú ya dominas:

Acción del usuario en el formularioSQL que se ejecuta por detrás
Llena campos y pica "Guardar nuevo"INSERT INTO alumnos (...) VALUES (...)
Modifica un dato y pica "Actualizar"UPDATE alumnos SET ... WHERE matricula = ...
Pica "Eliminar registro"DELETE FROM alumnos WHERE matricula = ...
Abre el formulario y ve datos existentesSELECT ... FROM alumnos WHERE matricula = ...

Un buen formulario además valida antes de ejecutar: campos obligatorios llenos (NOT NULL), matrícula no repetida (PRIMARY KEY), datos del tipo correcto. Las reglas que definiste al crear la tabla son el "contrato" que el formulario hace cumplir. Los gestores ofrecen herramientas para crearlos (Access los genera desde la tabla; phpMyAdmin da una pestaña Insertar; en la web se programan con HTML + PHP).

B · Reportes — consultas con presentación

Un reporte es una consulta preparada para entregarse: columnas con nombres legibles, datos ordenados y totales calculados. Tres ingredientes nuevos:

1 · AS — renombrar columnas (alias)

🗄 SistemaAdministracionEstudiantil › alumnos
SELECT nombre AS Alumno, promedio AS 'Promedio General'
FROM alumnos;

2 · ORDER BY — ordenar resultados

-- ASC: menor a mayor (es el orden por defecto) · DESC: mayor a menor
SELECT nombre, promedio FROM alumnos
ORDER BY promedio DESC;

3 · Funciones de resumen + GROUP BY

Las funciones de agregación convierten muchas filas en un solo dato: COUNT(*) cuenta, AVG() promedia, SUM() suma, MAX()/MIN() dan extremos. Con GROUP BY el resumen se calcula por categoría:

🗄 SistemaAdministracionEstudiantil › alumnos
-- ¿Cuántos alumnos y qué promedio tiene CADA grupo?
SELECT grupo,
       COUNT(*) AS total_alumnos,
       ROUND(AVG(promedio), 2) AS promedio_general
FROM alumnos
GROUP BY grupo
ORDER BY promedio_general DESC;
La regla del GROUP BY

Si usas una función de resumen junto con una columna normal (como grupo), esa columna debe ir en el GROUP BY. Léelo así: "agrupa las filas por grupo, y para cada grupo cuenta y promedia". ROUND(x, 2) es formato: redondea a 2 decimales para que el reporte se vea profesional.

El combo profesional

Vista + reporte trabajan en equipo: creas una vista con el JOIN complejo, y el reporte es un SELECT sencillo sobre esa vista con AS, ORDER BY y funciones de resumen. Eso harás en el ejemplo guiado.

Sección 4 · Ejemplo guiado

Del requerimiento al reporte, paso a paso

Vamos a resolver un caso real con el combo profesional: primero una vista que esconde la complejidad, luego un reporte sencillo encima de ella. El método de 4 pasos sigue siendo tu brújula.

PASO 1
¿Qué información se solicita?
PASO 2
¿En qué tablas se encuentra?
PASO 3
¿Conviene guardarla como vista?
PASO 4
¿Qué formato pide el reporte?

📋 Requerimiento real

La coordinación académica solicita: "Quiero poder consultar en cualquier momento las boletas del parcial — nombre del alumno, materia y calificación — sin tener que pedirle a sistemas que arme la consulta cada vez. Y para la junta, un reporte ordenado de mayor a menor calificación con encabezados legibles."

Análisis con el método

  1. ¿Qué información? Nombre, materia y calificación → columnas de tres tablas distintas.
  2. ¿En qué tablas? alumnos, materias y calificaciones, unidas por sus llaves (lo dominas del RA 2.1).
  3. ¿Vista? Sí: "poder consultarla en cualquier momento sin armarla cada vez" es la definición de una vista.
  4. ¿Formato? Encabezados legibles (AS) y orden de mayor a menor (ORDER BY ... DESC).

Paso A · Crear la vista (una sola vez)

🗄 SistemaAdministracionEstudiantil › vista_boletas
CREATE VIEW vista_boletas AS
SELECT a.nombre, a.apellido_paterno, m.nombre_materia, c.calificacion
FROM calificaciones c
INNER JOIN alumnos  a ON c.matricula  = a.matricula
INNER JOIN materias m ON c.id_materia = m.id_materia;
Línea¿Qué hace?
CREATE VIEW vista_boletas ASBautiza la consulta. Todo lo que sigue al AS queda guardado bajo ese nombre.
SELECT ... INNER JOIN ...La consulta del RA 2.1, con sus dos puentes: matricula y id_materia. Se escribe UNA vez en la vida.

Paso B · El reporte para la junta (cada vez que se necesite)

🗄 SistemaAdministracionEstudiantil › vista_boletas
SELECT nombre AS Alumno,
       nombre_materia AS Materia,
       calificacion AS 'Calificación'
FROM vista_boletas
ORDER BY calificacion DESC;

Observa el poder del combo: el reporte ya no menciona JOIN ni tres tablas — consulta la vista como si fuera una tabla simple, renombra columnas y ordena. Si mañana piden "solo calificaciones de 9 en adelante", basta agregar WHERE calificacion >= 9.

Sección 8 · Práctica libre

Simulador SQL · tu gestor en el navegador

Esto no es una imitación: es un motor de bases de datos real (SQLite) corriendo dentro de tu navegador, con la base SistemaAdministracionEstudiantil ya cargada. Escribe SQL, ejecútalo y observa resultados y errores reales. Nada de lo que hagas aquí afecta tus actividades entregadas, y puedes reiniciar la base cuando quieras.

El simulador usa SQLite: 3 pequeñas diferencias con MySQL

1) Para fechas usa datetime('now') en lugar de NOW(). 2) El autoincremental se escribe AUTOINCREMENT (sin guion bajo). 3) Los disparadores envuelven su acción entre BEGIN y END;. Todo lo demás de este cuaderno (SELECT, WHERE, INSERT, UPDATE, DELETE, INNER JOIN) funciona idéntico.
